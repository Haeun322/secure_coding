# 중고거래 플랫폼 개발 보고서

작성일: 2026-07-24

이 문서는 중고거래 플랫폼을 처음부터 만들면서 어떤 순서로 무엇을 고민했는지,
그리고 보안 관점에서 어디가 위험할 수 있었고 그걸 어떻게 막았는지를 정리한 글이다.
과제 요구대로 요구사항 분석 → 설계 → 구현 → 체크리스트/테스트 → 유지보수 순서로 적었고,
개발하면서 실제로 걸렸던 부분들을 최대한 솔직하게 남겼다.

---

## 1. 무엇을 만들었나

한마디로 하면 "번개장터/당근마켓의 축소판"이다. 사용자가 가입해서 상품을 올리고,
다른 사람 상품을 보고, 판매자와 1:1로 대화하고, 앱 안에서 돈을 주고받을 수 있다.
악성 유저나 이상한 상품은 신고할 수 있고, 관리자가 전체를 통제한다.

요구사항을 기능 단위로 끊으면 이렇게 된다.

| 요구사항 | 구현한 것 |
|---|---|
| 가입 | 아이디/비밀번호 회원가입, 로그인/로그아웃, 프로필/비밀번호 변경 |
| 상품 등록·조회 | 상품 CRUD, 이미지 업로드, 목록/상세 |
| 사용자 간 소통 | 1:1 메시지(대화방 목록 + 스레드) |
| 악성 유저·상품 차단 | 신고 기능, 신고 누적 시 자동 숨김, 관리자 수동 차단 |
| 송금 | 지갑 잔액, 사용자 간 이체, 거래 내역 |
| 검색 | 제목/설명 키워드 검색 |
| 관리자 | 사용자·상품·신고·송금 전체 조회 및 차단/해제 |

기술 스택은 Python 3.14 + Flask 3.1 + SQLite로 잡았다. 이유는 아래 설계 부분에 적었다.

---

## 2. 요구사항 분석

### 2.1 기능 요구사항

과제에 적힌 7가지를 그대로 기능으로 옮겼다. 다만 "송금"과 "차단"은 그냥 만들면
바로 사고가 나는 기능이라 처음부터 별도로 신경 써야겠다고 생각했다.

- 송금: 돈이 오가는 곳이라 잔액이 음수가 되거나, 동시에 두 번 눌렀을 때 이중으로
  빠져나가는 상황을 반드시 막아야 한다.
- 차단: 차단당한 사용자가 세션 쿠키만 들고 계속 돌아다니면 차단이 무의미하다.
  차단이 "즉시" 먹혀야 한다.

### 2.2 보안 요구사항 (비기능)

이 과제는 사실상 "기능을 만들되 취약점을 만들지 마라"가 핵심이다. 그래서 기능 명세와
별개로, 웹 애플리케이션에서 흔히 터지는 항목을 미리 목록으로 뽑아 두고 개발 내내
이 목록을 기준으로 코드를 봤다. (OWASP Top 10을 참고해서 이 프로젝트에 해당하는 것만 추렸다.)

- 인증/세션: 비밀번호 평문 저장 금지, 세션 고정·탈취 대비
- 접근 통제: 남의 리소스(상품, 대화, 지갑)에 손대지 못하게
- 인젝션: SQL Injection, XSS
- CSRF: 상태를 바꾸는 요청은 전부 토큰 검증
- 파일 업로드: 위장 파일/실행 파일 차단
- 무차별 대입(brute force): 로그인 시도 제한
- 민감정보 노출: 에러 스택, 비밀번호 해시, 잔액 등

## 3. 시스템 설계

### 3.1 언어·프레임워크 선택

Flask를 골랐다. 이유가 몇 개 있다.

첫째, 이 과제의 목적이 "보안 처리를 직접 이해하고 적용하는 것"이라고 봤다.
Django처럼 CSRF·인증이 다 자동으로 되는 프레임워크를 쓰면 편하긴 한데, 정작
"내가 뭘 막고 있는지"가 잘 안 보인다. Flask는 최소한만 주기 때문에 CSRF나 세션,
접근 제어를 내가 손으로 붙이게 되고, 그 과정이 그대로 보고서 재료가 된다.

둘째, 의존성을 적게 가져가고 싶었다. 외부 패키지가 많을수록 공급망 취약점 표면이
넓어진다. 그래서 실제 런타임 의존성은 Flask와 Werkzeug 딱 두 개다. CSRF 토큰,
레이트리밋, 입력 검증은 표준 라이브러리(`secrets`, `hmac`, `re`)로 직접 짰다.

DB는 SQLite로 했다. 별도 서버 설치 없이 파일 하나로 돌아가서 채점/재현이 쉽고,
`sqlite3` 모듈이 파라미터 바인딩을 기본으로 지원해서 SQL Injection을 막기에도 충분하다.
규모가 커지면 PostgreSQL로 옮기면 되는데, 쿼리를 전부 파라미터 바인딩으로 짜뒀기 때문에
그 전환 비용은 크지 않다.

### 3.2 구조

애플리케이션 팩토리 + 블루프린트 구조로 나눴다. 기능별로 파일이 갈려 있어서
관리자 코드 보다가 송금 코드로 넘어가는 게 쉽다.

```
app/
  __init__.py      앱 조립(팩토리), 에러 핸들러, 관리자 부트스트랩
  config.py        환경변수 로딩, 세션 쿠키 옵션
  db.py            SQLite 연결(요청용/쓰기전용 분리)
  schema.sql       테이블 정의
  security.py      CSRF, 인증 데코레이터, 레이트리밋, 보안 헤더
  validators.py    입력값 검증 함수 모음
  blueprints/
    main / auth / products / chat / payments / reports / admin
  templates/       Jinja2 (자동 이스케이프)
  static/          style.css, app.js
tests/
  test_app.py      기본 기능·보안 테스트
  test_hardening.py 2차 검증 회귀 테스트
```

보안 관련 코드를 `security.py`, `validators.py` 두 곳에 몰아둔 건 의도적이다.
"검증은 여기서만 한다"는 규칙을 만들어 두면, 나중에 어떤 기능을 추가할 때도
같은 관문을 통과시키게 된다.

### 3.3 데이터 모델

금액은 전부 정수(원)로 저장한다. 처음엔 실수로 할까 하다가, 0.1 + 0.2 = 0.30000000004
같은 부동소수점 오차가 돈 계산에 끼면 안 되겠다 싶어서 정수로 고정했다. 잔액 컬럼에는
`CHECK (balance >= 0)` 제약을 걸어서, 애플리케이션 로직이 뚫리더라도 DB 레벨에서
음수 잔액을 거부하도록 했다(다층 방어).

사용자·상품에는 `status` 컬럼을 둬서 active/blocked를 표현한다. 차단은 행을 지우는 게
아니라 상태를 바꾸는 방식이다. 지워버리면 신고 이력이나 거래 내역이 같이 날아가서
나중에 확인이 안 된다.

## 4. 구현

기능별로 어떻게 만들었는지, 특히 보안적으로 신경 쓴 부분 위주로 적는다.

### 4.1 회원가입 / 로그인

비밀번호는 Werkzeug의 `generate_password_hash`로 해싱해서 저장한다. salt가 자동으로
붙고, 검증은 `check_password_hash`로 한다. 평문은 어디에도 남기지 않는다.

로그인이 성공하면 `session.clear()`로 세션을 비우고 새로 시작한다. 세션 고정(session
fixation) 공격을 막기 위해서다. 로그인 전에 공격자가 심어둔 세션 ID가 로그인 후에도
그대로 쓰이면 안 되니까, 로그인 시점에 세션을 갈아끼운다.

로그인 실패는 아이디가 틀렸든 비밀번호가 틀렸든 "아이디 또는 비밀번호가 올바르지 않습니다"
하나로 통일했다. 어느 쪽이 틀렸는지 알려주면 계정 존재 여부가 새어나간다.

### 4.2 상품 + 이미지 업로드

이미지 업로드가 은근히 위험한 지점이라 검증을 3단으로 걸었다.

1. 확장자 화이트리스트 (png/jpg/gif/webp만)
2. 파일 앞부분 매직바이트 확인 — `.png`인데 내용이 실제 PNG인지 시그니처로 검사한다.
   브라우저가 보낸 Content-Type 헤더는 위조가 가능해서 믿지 않는다.
3. 저장 파일명은 `secrets.token_hex(16)`으로 새로 만든다. 사용자가 준 파일명은
   저장 경로에 절대 반영하지 않는다. `../../evil` 같은 경로 순회를 원천 차단하려는 것.

파일 서빙은 `send_from_directory`로 한다. 이 함수가 지정 폴더 밖으로 나가는 경로를
자체적으로 막아준다.

### 4.3 검색

검색어를 LIKE 쿼리에 바로 넣으면 두 가지가 문제다. 하나는 SQL Injection, 하나는
LIKE 와일드카드(`%`, `_`) 오남용이다. 앞은 파라미터 바인딩(`?`)으로 막고, 뒤는
사용자 입력의 `%`, `_`, `\`를 이스케이프한 뒤 `ESCAPE '\'`를 붙여서 처리한다.

### 4.4 1:1 메시지

대화 조회 쿼리에서 "내 id"를 절대 조건에서 빼지 않는 걸 원칙으로 잡았다. 스레드를
불러올 때 `(내가 보냄 AND 상대가 받음) OR (상대가 보냄 AND 내가 받음)` 조건이 항상
들어간다. URL의 상대방 id만 바꿔서 남의 대화를 훔쳐보는 시도(IDOR)를 막기 위해서다.

### 4.5 송금 (제일 공들인 부분)

돈이 오가는 곳이라 가장 방어적으로 짰다.

- 금액은 양의 정수만. 문자열/실수/음수/과대값은 검증 단계에서 전부 걸러낸다.
- 자기 자신에게 송금 금지, 차단된 사용자에게 송금 금지.
- **비밀번호 재확인**: 송금 폼에서 비밀번호를 한 번 더 받는다. 세션이 잠깐 탈취돼도
  비밀번호 없이는 돈을 못 빼가게 하는 재인증 장치다.
- **원자적 처리**: 잔액 확인 → 차감 → 증가 → 기록을 하나의 트랜잭션으로 묶는다.
  `BEGIN IMMEDIATE`로 트랜잭션 시작 시점에 쓰기 잠금을 잡아서, 두 개의 송금이 동시에
  같은 잔액을 읽고 각각 빼가는 경쟁 조건(이중지불)을 막는다.
- 그리고 마지막 안전망으로 DB의 `CHECK (balance >= 0)`가 있다. 로직에 구멍이 나도
  잔액이 음수가 되는 UPDATE는 DB가 거부하고, 트랜잭션 전체가 롤백된다.

송금용 연결은 요청용 연결(`g.db`)과 분리했다. 자동커밋 모드로 따로 열어서
`BEGIN IMMEDIATE ~ COMMIT` 경계를 내가 직접 통제하도록 했다. `busy_timeout`을 줘서
동시 요청이 겹치면 에러 대신 잠깐 기다리게 했다.

### 4.6 신고 / 차단

사용자는 다른 유저나 상품을 신고할 수 있다. 같은 대상을 중복 신고하는 건
DB의 UNIQUE 제약으로 막았다. 서로 다른 사람에게서 신고가 3건 이상 쌓이면 관리자가
보기 전이라도 자동으로 숨김(blocked) 처리해서 피해가 번지는 걸 늦춘다. 단, 관리자
계정은 자동 차단 대상에서 빼놨다.

### 4.7 관리자

관리자 전용 라우트는 전부 `admin_required` 데코레이터를 건다. 일반 사용자가 URL을
직접 쳐서 들어와도 403이다. 관리자는 사용자/상품/신고/송금 내역을 다 보고, 차단과
해제를 할 수 있다. 상태를 바꾸는 동작은 예외 없이 POST + CSRF 토큰으로만 가능하다.

첫 관리자 계정은 환경변수(`ADMIN_USERNAME`/`ADMIN_PASSWORD`)로 최초 실행 때 한 번
자동 생성된다. 코드에 비밀번호를 박아두지 않으려는 것이다.

## 5. 개발 중 확인한 보안 약점과 대응

여기가 이 보고서의 핵심이다. 처음에 대충 짰다가 "이거 위험한데" 하고 고친 것들,
그리고 처음부터 위험을 알고 설계로 막은 것들을 함께 적는다. 개선 전/후가 있는 건
같이 남겼다.

### (1) SQL Injection

- **위험**: 검색이나 로그인에서 사용자 입력을 문자열로 이어붙여 쿼리를 만들면,
  `' OR '1'='1` 같은 입력으로 인증을 우회하거나 데이터를 통째로 빼갈 수 있다.
- **대응**: 프로젝트의 모든 쿼리를 파라미터 바인딩(`?`)으로만 작성한다는 규칙을 세웠다.
  문자열 포매팅(`f"... {x}"`)으로 SQL을 만드는 코드는 한 곳도 두지 않았다. 검색의
  LIKE는 와일드카드까지 이스케이프했다.
- **검증**: `test_search_sql_injection_is_safe`에서 `' OR '1'='1`을 넣어도 200으로
  정상 처리되는 걸 확인했다.

### (2) XSS (교차 사이트 스크립팅)

- **위험**: 상품 설명이나 메시지에 `<script>`를 넣으면, 그 페이지를 보는 다른 사용자
  브라우저에서 스크립트가 실행될 수 있다.
- **대응**: Jinja2 템플릿의 자동 이스케이프를 그대로 쓴다. 어디에서도 `|safe` 필터나
  `Markup`으로 이스케이프를 끄지 않았다. 줄바꿈을 살려야 하는 상품 설명은 HTML을
  허용하는 대신 CSS `white-space: pre-wrap`으로 처리했다. 여기에 더해 응답에
  Content-Security-Policy 헤더를 붙여서, 설령 뭔가 새어나가도 인라인 스크립트나
  외부 스크립트가 실행되지 않도록 이중으로 막았다.
- **검증**: `test_xss_payload_is_escaped`에서 `<script>`가 `&lt;script&gt;`로
  이스케이프되는 걸 확인했다.

### (3) CSRF

- **위험**: 로그인된 사용자를 악성 사이트로 유인해서, 사용자도 모르게 송금이나 상품
  삭제 같은 요청을 대신 보내게 만들 수 있다.
- **대응**: 세션마다 CSRF 토큰을 발급하고, 상태를 바꾸는 모든 요청(POST/PUT/PATCH/DELETE)에서
  토큰을 검증한다. 검증은 `before_request`에 걸어서 빠뜨리는 라우트가 없게 했고,
  비교는 `hmac.compare_digest`로 상수 시간에 한다(타이밍 공격 방지). 세션 쿠키에는
  `SameSite=Lax`도 걸어서 한 겹 더 막는다.
- **검증**: `test_post_without_csrf_is_forbidden`, `test_post_with_bad_csrf_is_forbidden`
  에서 토큰이 없거나 틀리면 403이 나는 걸 확인했다.

### (4) 접근 통제 미흡 (IDOR / 권한 상승)

- **위험**: `/products/1/edit` 처럼 URL의 id만 바꿔서 남의 상품을 수정하거나, 일반
  사용자가 `/admin/`에 직접 접근하는 경우.
- **대응**: 상품 수정·삭제에서 `seller_id == 현재 사용자` 소유권을 확인한다. 대화·지갑도
  본인 것만 조회된다. 관리자 기능은 전부 `admin_required`. 그리고 세션에 저장된 role을
  믿지 않고, 매 요청마다 DB에서 사용자를 다시 읽어 role/status를 확인한다.
- **검증**: `test_cannot_edit_others_product`(403), `test_normal_user_cannot_access_admin`(403).

### (5) 차단이 즉시 안 먹히는 문제

- **처음 만들었을 때**: 로그인하면 세션에 사용자 정보를 통째로 담아두고 그걸 재사용하려
  했다. 그런데 이러면 관리자가 어떤 사용자를 차단해도, 그 사용자는 이미 발급된 세션으로
  계속 돌아다닌다. 차단이 실질적으로 무의미해진다.
- **바꾼 것**: 세션에는 `user_id`만 저장하고, 매 요청마다 그 id로 DB를 다시 조회한다.
  조회 결과 status가 blocked거나 계정이 사라졌으면 그 자리에서 세션을 비운다. 덕분에
  차단은 다음 요청부터 바로 적용된다.
- **검증**: `test_admin_can_block_and_unblock_user`에서 차단 후 로그인 시도가
  "차단된 계정" 메시지로 막히는 걸 확인했다.

### (6) 무차별 대입(brute force) 로그인

- **위험**: 비밀번호를 자동으로 계속 넣어보며 뚫는 공격.
- **대응**: 로그인 시도를 DB(`login_attempts`)에 기록하고, 15분 창 안에서 실패가 5회를
  넘으면 계정명 기준·IP 기준 양쪽으로 잠시 막는다(429). 성공하면 그 기록을 지운다.
- **검증**: `test_login_rate_limit`에서 실패를 반복하면 429가 나오는 걸 확인했다.

### (7) 사용자 열거 + 로그인 타이밍

- **위험**: 존재하지 않는 아이디는 비밀번호 검증을 건너뛰어 응답이 빨리 돌아온다.
  이 시간 차이로 어떤 아이디가 실제로 있는지 알아낼 수 있다.
- **대응**: 아이디가 없어도 미리 만들어 둔 더미 해시로 `check_password_hash`를 똑같이
  돌려서, 존재 여부와 상관없이 비슷한 시간이 걸리게 했다. 실패 메시지도 하나로 통일했다.

### (8) 파일 업로드로 악성 파일 심기

- **위험**: 이미지인 척하는 스크립트/실행 파일을 올려서 서버에서 실행시키거나, 다른
  사용자에게 내려보내는 경우. 사용자 파일명을 그대로 쓰면 경로 순회도 가능하다.
- **대응**: (4.2에 적은 대로) 확장자 화이트리스트 + 매직바이트 검증 + 무작위 파일명 +
  용량 제한(5MB). `X-Content-Type-Options: nosniff`로 브라우저의 MIME 스니핑도 껐다.

### (9) 송금 경쟁 조건 / 음수 잔액

- **위험**: 잔액 1,000원인 사람이 동시에 1,000원짜리 송금 두 건을 보내면, 둘 다 잔액을
  1,000원으로 읽고 각각 통과시켜 -1,000원이 될 수 있다(이중지불).
- **대응**: (4.5에 적은 대로) `BEGIN IMMEDIATE` 트랜잭션으로 잔액 확인과 이체를 원자적으로
  묶고, DB `CHECK (balance >= 0)`를 최후 방어선으로 뒀다. 비밀번호 재확인도 요구한다.
- **검증**: `test_transfer_requires_password_and_balance`에서 잔액 초과/비밀번호 오류
  송금이 거부되는 걸, `test_cannot_transfer_to_self`/`test_negative_amount_rejected`에서
  자기 송금·음수 금액이 막히는 걸 확인했다.

### (10) 민감정보 노출

- **위험**: 디버그 모드의 스택트레이스, 남의 프로필에서 잔액/비밀번호 해시가 보이는 것.
- **대응**: 운영에서는 `debug=False`. 커스텀 에러 페이지(403/404/413/429)로 내부 정보를
  감춘다. 공개 프로필 쿼리는 필요한 컬럼만 뽑고 balance/password_hash는 애초에 select 하지
  않는다. 비밀키는 코드가 아니라 환경변수(.env)로 주입하고 `.gitignore`로 저장소에서 뺀다.

### (11) 오픈 리다이렉트

- **위험**: 로그인 후 이동할 `next` 파라미터에 외부 URL을 넣어 피싱 사이트로 튕기는 경우.
- **대응**: `next`가 `/`로 시작하는 내부 경로인지 확인하고, 스킴이나 호스트가 붙어 있으면
  무시하고 홈으로 보낸다.

### (12) 보안 헤더 부재

- **대응**: 모든 응답에 CSP, `X-Frame-Options: DENY`(클릭재킹 방지),
  `X-Content-Type-Options: nosniff`, `Referrer-Policy`, `Permissions-Policy`를 붙였다.
  세션 쿠키는 HttpOnly + SameSite=Lax, 운영에서는 Secure까지 켠다.

## 6. 2차 정밀 검증에서 추가로 잡은 것들

1차로 기능을 다 만든 뒤, "정말 버그가 없나"를 다시 확인하려고 코드를 처음 보는 눈으로
다시 훑었다. 이때는 인증/세션, 송금 동시성, 접근 통제, 인젝션/CSP를 각각 따로 파고들어
독립적으로 점검했고, 의심 가는 건 실제로 재현해 보고 나서야 확정했다. 여기서 1차 때
놓친 실제 결함이 여러 개 나왔다. 등급은 실제 영향 기준이다.

### (13) 차단된 상품을 소유자가 수정으로 되살릴 수 있었음 — **높음**

- **문제**: 상품 수정(`edit`)이 소유권만 확인하고 현재 상태는 보지 않았다. 관리자나
  자동 모더레이션으로 `blocked` 된 상품이라도, 주인이 수정 폼을 그냥 저장하면 상태가
  `active`로 돌아갔다. 신고 3건으로 자동 차단된 사기 상품을 올린 사람이 이걸 무한 반복하면
  모더레이션 자체가 무력화된다.
- **고침**: 편집·삭제 진입 시 `status == 'blocked'`면 소유자는 손대지 못하게 막았다
  (관리자만 삭제 가능). 상품이 사라지면 그 상품을 향한 신고도 함께 정리해서 관리자
  큐가 오염되지 않게 했다.
- **검증**: `test_owner_cannot_unblock_blocked_product_via_edit`,
  `test_reports_cleaned_on_product_delete`.

### (14) 누구나 아무 계정이나 잠글 수 있는 로그인 잠금 DoS — **높음**

- **문제**: 로그인 실패 제한을 "계정명" 기준으로만 걸었더니, 공격자가 남의 계정명으로
  일부러 5번 실패시키면 그 사람(심지어 관리자)이 15분간 로그인 못 하게 됐다. 잠금 장치가
  거꾸로 공격 도구가 된 셈이다.
- **고침**: 잠금 축을 `IP` 와 `계정명+IP` 조합으로 바꿨다. 이러면 공격자가 잠그는 건
  "그 공격자 IP에서의 그 계정" 뿐이라, 피해자는 자기 IP에서 정상적으로 로그인된다.
  한 IP가 여러 계정을 노리는 무차별 대입은 IP 축이 잡는다.
- **검증**: `test_account_lockout_does_not_affect_victim_own_ip`.

### (15) 송금 동시성(이중지불) 정밀 확인 — **확인/보강**

- 1차 설계(`BEGIN IMMEDIATE` + `CHECK`)가 실제로 맞는지 스레드 10개로 동시에 같은
  잔액을 털어보는 테스트를 돌렸다. 결과는 성공 정확히 1건, 잔액 0, 음수 없음, 원장 1건으로
  의도대로였다. 다만 SQLite 기본 저널 모드에서는 읽기 연결과 쓰기 연결이 서로를 막아
  드물게 "database is locked"가 날 수 있어서, **WAL 모드와 `busy_timeout`을 켜서** 그
  잠재적 교착을 없앴다.
- **검증**: `test_concurrent_transfers_no_double_spend`(스레드 10개 동시 송금).

### (16) 차단된 계정인지 아닌지가 새어나감 — **중간**

- **문제**: 로그인 실패 시 차단 계정이면 "차단된 계정입니다"를 보여줬는데, 이게 비밀번호가
  틀려도 떴다. 즉 비밀번호를 몰라도 어떤 아이디가 존재하고 차단됐는지 알아낼 수 있었다.
- **고침**: 차단 안내는 비밀번호가 맞은 경우(=진짜 본인)에만 보여주고, 그 외 실패는 전부
  동일한 일반 메시지로 응답한다.
- **검증**: `test_blocked_account_not_enumerable_with_wrong_password`.

### (17) 오픈 리다이렉트 역슬래시 우회 — **중간**

- **문제**: `next` 검사가 `/\evil.com` 같은 역슬래시 섞인 값을 통과시켰다. 브라우저가 이걸
  `//evil.com`으로 정규화해서 외부로 튕길 수 있었다.
- **고침**: 역슬래시·제어문자·`//` 시작을 모두 거르도록 검사를 강화했다.
- **검증**: `test_open_redirect_backslash_blocked`.

### (18) CSP와 인라인 스타일/스크립트가 서로 안 맞았음 — **중간**

- **문제**: CSP를 `script-src 'self'` / `style-src 'self'`로 엄격하게 걸어놓고, 정작
  템플릿에는 인라인 `style="..."` 속성과 인라인 `onsubmit=` 핸들러가 있었다. 실제 브라우저는
  이걸 CSP 위반으로 막기 때문에, 스타일이 깨지고 삭제 확인창이 안 뜨는 상태였다.
- **고침**: 인라인 스타일 14곳을 전부 CSS 클래스로 옮기고, 인라인 핸들러는 외부 `app.js`의
  `data-confirm` 방식으로 바꿨다. 덕분에 `'unsafe-inline'`을 열지 않고도 화면이 정상 동작한다.
  더불어 HTTPS일 때 HSTS 헤더를, CSP에 `object-src 'none'`을 추가했다.
- **검증**: `test_csp_and_hsts_headers`.

### (19) 금액 파싱이 사용자 의도와 다른 값을 만들 수 있었음 — **중간**

- **문제**: 금액에서 콤마를 무조건 제거하다 보니 `1,2,3`이 123이 되고, 전각 숫자 `１２３`도
  int로 먹혀서 123이 됐다. 오타가 다른 금액으로 조용히 바뀔 수 있었다.
- **고침**: ASCII 숫자만 허용하고, 콤마는 올바른 천단위(`1,000`)일 때만 인정하도록 파서를
  다시 짰다. 초대형 자릿수 입력도 길이 제한으로 먼저 거른다.
- **검증**: `test_amount_parser_strictness`.

### (20) 그 밖의 보강

- **데모 충전 상한**: `topup`은 실제 PG 연동이 없는 데모라, 계정 총 잔액에 상한(1천만 원)을
  둬서 무한정 돈을 찍지 못하게 했다. (실서비스에선 결제 승인 검증으로 대체 필요 — 코드에 명시)
- **프록시 뒤 IP 복원**: 리버스 프록시 뒤에 둘 때 `TRUST_PROXY_HOPS`로 신뢰 홉 수를
  지정하면 `X-Forwarded-For`로 실제 클라이언트 IP를 복원한다(레이트리밋 정확도).
- **비밀번호 정책**: 흔한 비밀번호 차단 목록과 "아이디를 비밀번호에 포함 금지"를 추가.
- **로그 테이블 관리**: `login_attempts`가 무한히 커지지 않도록 오래된 기록을 가끔 정리.
- **신고 재차단 방지**: 관리자가 차단을 풀면 남아 있던 열린 신고를 처리 완료로 정리해서,
  신고 하나 더 들어왔다고 곧바로 다시 자동 차단되는 문제를 없앴다.

> 참고로 이 2차 검증에서 SQL Injection, XSS, 경로 순회, 권한 없는 관리자 접근,
> mass assignment는 별도로 재점검한 결과 문제가 없음을 확인했다.

### 7.2 테스트

`pytest`로 통합·단위 테스트 32개를 작성했다. 위 체크리스트 항목이 실제로 동작하는지를
자동으로 검증하도록 했고, 매 테스트는 임시 폴더에 새 DB를 만들어 서로 간섭하지 않게 했다.
메모리 기반 레이트리밋 버킷도 테스트마다 초기화한다.

```
$ python -m pytest tests/ -q
................................                                         [100%]
32 passed
```

테스트는 크게 세 부류다. 기능이 되는지(가입/로그인/상품/송금/검색/페이지네이션),
막아야 할 게 막히는지(CSRF 없는 POST, 남의 상품 수정, SQLi, XSS, 자기 송금, 음수/이상 금액,
관리자 무단 접근, 모더레이션 우회, 오픈 리다이렉트, 계정 열거), 그리고 정책·동시성이 도는지
(레이트리밋, 차단 후 로그인 불가, 계정 잠금 DoS 회피, 스레드 10개 동시 송금에서 이중지불 0건).

`tests/test_app.py`가 기본 기능·보안을, `tests/test_hardening.py`가 2차 검증에서
잡은 결함들의 회귀 방지를 담당한다.

## 8. 유지보수

만들고 끝이 아니라 계속 굴러가야 하니, 유지보수 관점에서 몇 가지를 정리해 뒀다.

- **설정 분리**: 비밀키·관리자 계정·쿠키 옵션이 전부 환경변수라 코드를 건드리지 않고
  환경만 바꿔서 운영/개발을 오갈 수 있다.
- **검증 일원화**: 입력 검증이 `validators.py`에, 보안 장치가 `security.py`에 모여 있다.
  기능을 추가할 때 같은 함수를 재사용하면 되니, 새 코드가 기존 방어를 우회할 여지가 적다.
- **테스트가 회귀 안전망**: 리팩터링하거나 기능을 더할 때 `pytest`를 돌리면 기존 보안
  동작이 깨졌는지 바로 잡힌다.
- **DB 이관 여지**: 쿼리를 전부 파라미터 바인딩으로 짜서, SQLite에서 PostgreSQL 등으로
  옮길 때 부담이 적다.
- **로그**: 로그인 시도는 `login_attempts` 테이블에 남아서 이상 징후를 사후에 볼 수 있다.

### 앞으로 개선하면 좋을 것 (한계)

솔직하게 남은 것도 적는다.

- 메모리 기반 레이트리밋은 단일 프로세스 기준이다. 워커를 여러 개 띄우면 Redis 같은
  공용 저장소로 옮겨야 정확해진다.
- 실제 결제(PG) 연동은 없다. 지금 "충전"은 데모용이고, 진짜 서비스라면 결제 대행사의
  승인 결과를 검증한 뒤에만 잔액을 올려야 한다.
- 이메일 인증, 비밀번호 재설정, 2단계 인증은 범위 밖이라 넣지 않았다.
- 운영 배포 시에는 개발 서버 대신 waitress/gunicorn 같은 WSGI 서버 뒤에 두고,
  HTTPS를 반드시 적용해야 한다(그래야 Secure 쿠키가 의미를 가진다).

## 9. 생성형 AI 활용

과제 안내대로 개발 과정에서 생성형 AI를 적극적으로 썼다. 주로 쓴 방식은 이렇다.

- 보안 항목을 빠뜨리지 않도록 OWASP 기준의 점검 목록을 함께 만들고, 각 기능마다
  "여기서 뚫릴 수 있는 지점"을 짚어보는 데 활용했다.
- 송금 트랜잭션의 경쟁 조건처럼 놓치기 쉬운 부분을 리뷰받아 `BEGIN IMMEDIATE`와
  `CHECK` 제약의 조합으로 다듬었다.
- 2차 정밀 검증(6장)에서는 인증/세션, 송금 동시성, 접근 통제, 인젝션/CSP를 각각
  따로 파고드는 방식으로 코드를 독립적으로 교차 리뷰했다. 이 과정에서 1차 때 놓쳤던
  실제 결함들(차단 상품 되살리기, 계정 잠금 DoS, 계정 열거, CSP-인라인 불일치 등)을
  찾아냈고, 각 지적은 실제로 재현해 확인한 뒤에만 수정으로 반영했다.

다만 최종 코드는 직접 읽고 이해한 뒤 반영했고, 지적받은 것도 그대로 믿지 않고 재현
테스트로 확인한 다음 고쳤다. AI가 제안한 것 중에서도 이 프로젝트에 맞지 않는 건 걸러냈다
(예: 무거운 외부 보안 라이브러리 도입은 의존성을 늘려서 제외, 업로드 이미지 재인코딩은
현재 방어(nosniff+확장자+매직바이트)로 충분해 보류).
